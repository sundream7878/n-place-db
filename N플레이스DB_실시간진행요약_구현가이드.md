# N플레이스DB 크롤러 실시간 진행요약 구현 가이드

## 0) 문서 운영 원칙

- 이 주제의 추가 요구사항/결정사항은 **별도 파일로 분산하지 않고 본 파일에 계속 통합**한다.
- 구매자에게 표시하는 모든 완료율/ETA 수치는 **추정치임을 명확히 표기**한다.
- **추정치는 시간이 지날수록(근거 데이터가 누적될수록) 정확도가 올라간다**는 문구를 반드시 유지한다.

---

## 1) 목적

구매자가 크롤러를 신뢰하려면, 단순 로그보다 **현재 어디까지 진행됐는지**를 한눈에 보여줘야 한다.  
이 문서는 N플레이스DB 수집기에서 바로 적용 가능한 **실시간 진행요약 표준안**을 제시한다.

핵심 원칙:
- `완료율`은 반드시 **N/M(현재 수집 개수/총 예상 개수)** 형태로 보여준다.
- `총 예상 개수(M)`는 추정치임을 명시한다.
- 수집이 진행될수록 근거 데이터가 쌓여 **추정 정확도가 점진적으로 올라가도록** 설계한다.

---

## 2) 구매자 화면에 보여줄 최소 구성

### 상단 4개 카드
- `예상 대상 수(추정)` = M
- `현재 수집 개수` = N
- `총 소요시간` = 시작 후 누적
- `예상 남은 시간` = ETA

### 하단 도표(1개만)
- `전체 완료율(추정): N/M (x.x%)`
- 진행바는 0.x%에서도 눈에 보이도록 최소 표시폭(예: 1.2%) 적용

### 보조 텍스트(2줄 권장)
- `개당 평균 소요시간: xx.x초/건`
- `현재 수집 구간: 지역/키워드/페이지 등 컨텍스트`

---

## 3) 용어 정의 (혼선 방지)

- `N (현재 수집 개수)`: 실제 저장 성공 건수 (중복 스킵/실패 제외)
- `M (총 예상 개수)`: 현재까지 관측 기반 추정 총량
- `완료율`: `N / M`
- `ETA`: 남은 예상 시간

중요:
- 스킵 개수, 실패 개수는 별도 로그/통계로 보여주되 `N`에는 포함하지 않는다.

---

## 4) N플레이스DB에서 M(총 예상 개수) 추정 로직

N플레이스는 대상 총량을 고정값으로 주지 않는 경우가 많으므로, 아래 **다층 추정**을 권장한다.

## 레벨 A: 직접 총량 확보 가능 시 (최우선)
- 검색/카테고리 응답에 total count가 있으면 그대로 사용
- 이 경우 신뢰도 높음(`confidence = high`)

## 레벨 B: 페이지 기반 추정 (일반적)
- 구간별(지역 x 키워드 x 카테고리)로:
  - 탐색한 페이지 수 `p_seen`
  - 페이지당 원시 수집 평균 `raw_per_page_avg`
  - dedup 후 순수 신규 비율 `new_ratio`
  - 예상 총 페이지 `p_est`
- 구간 예상량:
  - `segment_est = p_est * raw_per_page_avg * new_ratio`

`p_est` 산정:
- API/DOM에 마지막 페이지 근거가 있으면 반영
- 없으면 최근 k페이지의 신규 감소 추세로 포화점 추정
  - 신규 유입이 임계치 미만으로 떨어지는 시점을 종료점으로 본다

## 레벨 C: 샘플-외삽 추정 (초반)
- 시작 초반(근거 부족)에는 탐색 단위 일부만 샘플링
- 샘플당 신규 평균을 전체 단위 수로 외삽
- 탐색 진행률이 올라가면 B 방식으로 자동 전환

---

## 5) 추정치 안정화 규칙 (실무 핵심)

추정치가 튀면 구매자 신뢰가 떨어진다. 아래 보정이 중요하다.

- `M` 하한: 항상 `M >= N`
- 급격 점프 제한: 갱신 시 이전 M 대비 변화율 캡(예: ±20%)
- 지수평활:
  - `M_t = alpha * M_new + (1-alpha) * M_prev`
  - 권장 `alpha = 0.25 ~ 0.35`
- 표시 반올림: 천단위/백단위 반올림으로 흔들림 완화
- 갱신 주기: 매 건이 아닌 `5~20건 단위` 갱신 권장

---

## 6) ETA(예상 남은 시간) 계산

- 평균 처리시간:
  - `avg_sec = elapsed_sec / processed_success_count`
- 남은 건수:
  - `remain = max(0, M - N)`
- ETA:
  - `eta_sec = remain * avg_sec`

표시 형식:
- `예상 남은 시간: 4시간 10분 / 7시간 23분`
  - 분자: 남은 시간
  - 분모: `elapsed + eta`

---

## 7) 정확도 단계(신뢰도 뱃지) 표시 권장

추정치는 근거가 누적되며 정확해진다는 점을 화면에 명시한다.

- `초기 추정`: 진행률 < 10%
- `중간 추정`: 10% ~ 40%
- `안정 추정`: 40% 이상

예시 문구:
- `총 예상 개수는 추정치이며, 수집이 진행될수록 정확도가 올라갑니다.`

## 9) 운영 정책(구매자 신뢰 보호)

- 완료율/ETA가 계산 중일 때는 억지 숫자 대신 `계산 중...` 표기
- 네트워크/차단/재시도 시 상태 배너 명확히 표시
- 중단 후 재개 시:
  - 기존 N 유지
  - M 재추정 (이전 추정과 평활 결합)
- 로그에는 스킵/실패 사유를 남기되, 메인 완료율은 성공 기준 유지

## 12) 배포 및 유지보수 가이드 (2026-04-30 추가)

### 빌드 및 패키징 원칙
- **클린 빌드 필수**: 빌드 전 `dist` 폴더를 완전히 삭제하거나 이름을 변경하여 옛날 파일이 섞이지 않도록 한다.
- **폴더 잠금(PermissionError) 대처**: 
  - 윈도우 탐색기나 이전 실행 프로세스가 `dist` 폴더를 점유하고 있을 경우 빌드 실패 발생.
  - 해결: `taskkill /f /im python.exe` 등으로 관련 프로세스 종료 후 재시도. 
  - 지속될 경우 빌드 타겟 폴더명(예: `NPlace_DB`)을 살짝 변경하여 우회 빌드 후 최종 이름을 수정한다.

### 최종 패키지 구성 표준
- 유저에게 전달할 때는 **`NPlace-DB`** (하이픈 사용) 폴더로 전달한다.
- **핵심 구성 파일**:
  1. `NPlace-DB.exe`: 메인 프로그램 (보안 HWID 암호화 적용됨)
  2. `NPlace-DB-실행.bat`: 원클릭 실행 배치 파일
  3. `NPlace-DB_사용매뉴얼.pdf`: 최신 사용자 가이드
  4. `_internal`: 내부 라이브러리 폴더 (유저에게 노출 최소화)

### 버전 관리
- `config.py`의 `CURRENT_VERSION`을 최우선으로 관리한다.
- 대시보드 타이틀 옆에 표시되는 버전 정보를 통해 유저의 최신 버전 여부를 확인한다.
- 자동 업데이트 로직 적용 시 이 버전 정보를 기준으로 서버와 비교하여 업데이트 여부를 결정한다.
